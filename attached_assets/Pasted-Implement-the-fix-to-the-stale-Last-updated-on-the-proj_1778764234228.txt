Implement the fix to the stale "Last updated" on the /projects list.

ROOT CAUSE (from prior diagnose):
projects.updated_at is bumped only by direct project field edits 
(PATCH /api/projects/:id, cover-photo cleanup, share-token mint). 
Every other child mutation — photo upload, task complete, checklist 
update, report section add, timesheet clock-in, etc. — leaves the 
parent untouched. The /api/projects endpoint reads and sorts by this 
stale column.

FIX APPROACH: Option C from the diagnose — compute "last activity" 
on read in the storage method, no schema change, no fanout.

CHANGES:

1. server/storage.ts — getProjectsWithDetails:
   Rewrite the leading SELECT against projects to compute a 
   last_activity_at value via correlated subqueries, taking the 
   GREATEST of:
   - projects.updated_at (preserves manual-edit timestamps)
   - MAX(media.created_at) WHERE project_id = projects.id
   - MAX(tasks.updated_at) WHERE project_id = projects.id
   - MAX(checklists.updated_at) WHERE project_id = projects.id
   - MAX(checklist_items.updated_at) (joined via checklist.project_id 
     or however the schema relates them — verify the exact path)
   - MAX(reports.updated_at) WHERE project_id = projects.id
   - MAX(report_sections.updated_at) (joined via report.project_id)
   - MAX(media_annotations.updated_at) (joined via media.project_id)
   - MAX(comments.created_at) (joined via media.project_id — 
     comments have no updated_at per the diagnose)
   - MAX(time_entries.updated_at) WHERE project_id = projects.id

   Use COALESCE(..., '-infinity') for empty children so GREATEST 
   doesn't return NULL.

   Update the ORDER BY clause to use the computed last_activity_at 
   DESC, so the list is also sorted by recency of activity.

2. Return shape: keep the field name as updatedAt on the returned 
   object (alias the computed value) so the frontend at 
   client/src/pages/projects.tsx:306 doesn't need to change. The 
   semantic of updatedAt on this endpoint becomes "last activity" 
   while the column itself retains its original meaning — this is a 
   deliberate API-level aliasing.

3. NO frontend changes required. Verify by inspection.
