PART C — POST /api/invitations HANDLER UPDATE (server/routes.ts)

Extend body destructure to include:
- assignedProjectIds: number[] | undefined (defaults to [] if missing)

Validation logic:
- If role !== "restricted" AND assignedProjectIds.length > 0: 
  return 400 { error: "assigned_projects_invalid_role", message: 
  "Project assignments only apply to restricted role" }
- If role === "restricted" AND assignedProjectIds.length === 0: 
  allowed (assignment can happen post-acceptance via existing endpoints)
- All projectIds must belong to req.user's account. Validate via:
  SELECT id FROM projects WHERE id = ANY($1) AND account_id = $2
  If returned count !== assignedProjectIds.length: 400 { error: 
  "project_not_found", message: "One or more project IDs not found 
  or not accessible" }

On success, store the validated array in invitation.assigned_project_ids.

PART D — ACCEPTANCE FLOW

Find where invitations are consumed (the register-via-token or 
accept-invite endpoint — likely in 
server/replit_integrations/auth/routes.ts or a separate accept 
handler). After the new user row is created inside the existing 
acceptance transaction:

- For each projectId in invitation.assigned_project_ids, INSERT into 
  project_assignments (userId = new user's id, projectId, createdAt = 
  now()). Use the same tx — acceptance and assignments atomic.
- On any failure inside the tx, rollback so no orphaned assignments.

Log "[invite-accept] assigned user X to N projects from invitation Y" 
for verifiability.
