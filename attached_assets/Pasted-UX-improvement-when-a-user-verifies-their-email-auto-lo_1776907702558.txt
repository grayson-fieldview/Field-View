UX improvement: when a user verifies their email, auto-login them and redirect to the app so they land on the subscribe page (as an authenticated user). Currently they have to re-enter credentials after verifying, which is poor UX.

=== STEP 1: Backend — auto-login on verify ===

In server/replit_integrations/auth/replitAuth.ts, modify GET /api/verify-email:

1. Keep all existing validation logic (missing token, invalid, used, expired checks).

2. After the user is marked verified and the token is marked used, fetch the fully-updated user row:
   const [verifiedUser] = await db.select().from(users).where(eq(users.id, row.userId));

3. Call req.login(verifiedUser, (err) => {...}) to establish a session — mirror the same pattern already used in POST /api/register (before we removed it) or POST /api/login for how to call req.login and handle errors.

4. On successful session creation, return:
   { success: true, message: "Email verified successfully", user: { ...strip password... } }

5. If req.login fails (unlikely but possible), log the error and return the success response WITHOUT the user field. Frontend will handle this by redirecting to /login instead of /.

=== STEP 2: Frontend — verify-email page UX ===

In client/src/pages/verify-email.tsx, modify the success flow:

1. When GET /api/verify-email?token=... returns 200 with { success: true, user: {...} }:
   - Show a polished success state for 2 seconds:
     - Large green checkmark icon
     - Headline: "Email verified!"
     - Subtext: "Welcome to Field View. Taking you to your account..."
   - Set user state in whatever auth context/query the app uses (e.g., if you use React Query for the current user, invalidate the /api/auth/user query so it refetches).

2. If the response has { success: true } but NO user field (req.login failed):
   - Show the same success UI for 2 seconds
   - Redirect to /login instead of /

3. Error states unchanged (keep the existing "Invalid token" / "Token expired" / "Token used" handling).
