DIAGNOSE-FIRST then BUILD: pending invitations should reserve seats.

PROBLEM: Today, seat count only reflects ACCEPTED users. Admin can send 
unlimited pending invites regardless of seat availability. If multiple 
pending invites accept and total exceeds seat count, account is in a 
broken state (more active users than billed seats).

EXPECTED: When admin sends an invite, that invite should consume a seat 
slot immediately. The seat is released only when:
- Invite is canceled by admin
- Invite expires (current expiry rules apply)
- Invite is declined

When an invite is ACCEPTED, no seat-count change — it just transitions 
from "pending" to "active" within the same already-reserved slot.

PHASE 2 — DESIGN PROPOSAL (after investigation, before code):

Propose:
- New formula: usedSeats = COUNT(active users) + COUNT(non-expired 
  pending invites)
- Send invite endpoint: pre-check if usedSeats >= totalSeats → return 
  409 with { error: "no_seats_available", message, seatsAvailable: 0, 
  suggestion: "Purchase additional seats to invite more team members" }
- Accept invite endpoint: no seat-count change (already reserved); 
  just transition status
- Cancel invite endpoint: removes the row → seat freed automatically 
  via the new formula
- Expire-invite logic: if pending invite past expiry, NOT counted in 
  used seats (lazy check at query time, or background job — pick what 
  the codebase already does)
- UI updates:
  · Seats widget: "X of Y used (A active, B pending)" with breakdown
  · Invite User modal/button: disabled with tooltip if at capacity, 
    or show "Add seats first" CTA linking to billing
  · Frontend handles 409 gracefully (toast with the suggestion text)

Migration: existing 4-pending-invite scenario in Grayson's account
  · After ship, the seat count for his account jumps from 1 to 5
  · 5 > 3 = over capacity
  · Don't auto-cancel pending invites
  · Don't auto-revoke acceptances
  · Just display "5 of 3 used (over capacity)" — admin sees it, can 
    cancel pending invites OR purchase more seats
  · Grace state: invites can still be accepted (don't break inflight 
    flows for invitees), but new invites blocked until back at/under 
    capacity
