## 3. Wire up the invite flow

The existing invite flow in team.tsx works like this today:
- User clicks "Invite Team Member" → invite modal opens
- User fills in email + role → clicks "Send Invitation"
- sendInvite mutation fires → POST /api/invitations
- On success, invite modal closes

We're inserting a check between "click Send Invitation" and "fire 
POST /api/invitations":

  Current behavior:
    onClick={() => sendInvite.mutate()}

  New behavior:
    onClick={() => handleSendInvite()}

  Where handleSendInvite is a new function that:
    1. Reads seatStatus from the useQuery hook above
    2. If seatStatus is loading or errored → show toast "Cannot 
       check seat availability, try again" and bail
    3. If trial AND used >= 10 (seatStatus.trialMaxSeats) → show 
       toast "Trial accounts are limited to 10 seats. Upgrade to 
       add more." and bail
    4. If available > 0 → call sendInvite.mutate() as before. Done.
    5. If available === 0 AND not at trial cap → close invite modal, 
       set state to open the seat-add confirmation modal. Stash the 
       email + role temporarily so we can resume after seat growth.

The invite modal closes BEFORE the confirmation opens (rather than 
stacking dialogs). Two distinct steps: confirm seat add → then 
return to invite modal pre-filled and re-submit.

Alternative simpler version if state management gets too tangled: 
keep both modals open at once (confirmation modal stacks on top). 
Acceptable but slightly worse UX. Use your judgment.

## 4. Wire up the seat-add mutation

New useMutation:

  const addSeat = useMutation({
    mutationFn: async () => {
      const res = await apiRequest("POST", "/api/account/seats", {
        desiredCount: seatStatus.total + 1,
        expectedCurrent: seatStatus.total,
      });
      return res.json();
    },
    onSuccess: () => {
      queryClient.invalidateQueries({ queryKey: ["/api/account/seats"] });
      // After seat growth, proceed with the invite
      sendInvite.mutate();
    },
    onError: (error: Error) => {
      // 409 = concurrent change. Refresh and ask user to retry.
      if (error.message.includes("changed") || error.message.includes("refresh")) {
        queryClient.invalidateQueries({ queryKey: ["/api/account/seats"] });
        toast({ 
          title: "Seat count changed", 
          description: "Please try again.",
          variant: "destructive" 
        });
      } else {
        toast({ 
          title: "Failed to add seat", 
          description: error.message, 
          variant: "destructive" 
        });
      }
    },
  });

The seat-add confirmation dialog's onConfirm calls addSeat.mutate(). 
On success, the existing sendInvite mutation fires automatically.
