NEW PROJECT: Customer.io integration for lifecycle messaging.

CONTEXT
We're integrating Customer.io to handle lifecycle/marketing email (welcome series, activation drips, trial conversion, win-back). Transactional emails stay in Resend. Customer.io receives events from our server to trigger campaigns.

The email-sender audit PR currently in progress (separate task) is removing the welcome email from Resend. Customer.io will eventually take over welcome via a campaign triggered on the signed_up event we wire up here. Don't touch the Resend welcome — it's being handled in that other PR.

ARCHITECTURE DECISIONS (locked, don't re-litigate)
- Use Customer.io's official Node.js SDK: customerio-node
- Server-side only, no JS snippet on frontend
- Track API for identify + events; App API not needed this phase
- Region: US (CUSTOMERIO_REGION env var defaults to "us")
- Fire-and-forget with .catch() error logging — same pattern as the existing sendWelcomeEmail. Customer.io being slow or down NEVER blocks a user-facing response.
- All CIO calls live in a new server/services/customerio.ts module — mirrors server/services/email.ts structure
- When env vars are missing (dev environment), customerio service logs a warning once and becomes a no-op. Every call returns instantly without throwing.

ENV VARS (add to .env.example; I'll add real values in Vercel separately)

- CUSTOMERIO_TRACK_API_KEY
- CUSTOMERIO_APP_API_KEY (optional, prep for future phases)
