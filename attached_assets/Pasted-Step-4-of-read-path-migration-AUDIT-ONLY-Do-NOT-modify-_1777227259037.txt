# Context

We've migrated the backend read path (middleware + endpoint) and 
write path (webhook) for billing state to be flag-controlled via 
ACCOUNT_BILLING_ENABLED. Backend reads/writes route through 
getAccountBilling() / writeAccountBilling() helpers.

Now we need to verify the FRONTEND reads subscription state correctly 
under both flag states. Step 4 is an investigation — figure out what 
needs to change before we change anything.

## 3. Direct reads from user.subscriptionStatus / user.trialEndsAt /

For each match, cite the file:line and report what the code is doing

This is the critical question: does the frontend get billing state 
from the user object (which would be user-level data, populated by

or does it call /api/subscription separately?

If the frontend reads billing state from the user object, then we 
need to figure out where that user object is loaded and decide whether 
to:
(a) Update the loading endpoint to also include account-level billing

(b) Keep the user-level data on the loading endpoint, and have 
    components that care about billing call /api/subscription separately
