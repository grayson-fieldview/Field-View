These currently land in stdout only — Vercel logs and console output.

Sentry is already integrated (we saw sentry-environment=production in

captured events so we can:

- See failed-auth patterns in Sentry's dashboard
- Get notifications for unusual activity

## 3. Wire auth failures to Sentry

For each [auth-fail] log line, after the existing console.warn, also:

  Sentry.captureMessage("Auth failure", {
    level: "warning",
    tags: {
      reason: <string>,        // user_not_found / invalid_password / email_not_verified
      auth_provider: "local",  // (or whatever applies)
    },
    extra: {
      email,
      ip,
      ts,
    },
  });

Use captureMessage (not captureException) — these are not exceptions, 
they're security events. The level "warning" maps to a yellow indicator 
in Sentry's UI, not a red alert.

The tags vs extra distinction matters: tags are searchable/filterable 
in Sentry's UI; extra is metadata you see when you click into an event. 
Putting `reason` as a tag means you can filter "all auth failures with 
reason=invalid_password" in Sentry. Putting `email` as extra means it's 
visible but not part of the event grouping.

DO NOT put email or IP in tags — that would create unique events per 
email/IP and pollute the event list. Tags should be low-cardinality 
classifications.

## 4. Don't break the console.warn

Leave the existing console.warn in place. Sentry capture is additive — 
console logs are useful for local dev where Sentry might not be running, 
and for Vercel logs where we already grep them.

## 5. Sentry not initialized? Defensive handling

If Sentry isn't initialized (e.g., SENTRY_DSN not set in dev env), 
the captureMessage call should silently succeed without throwing. 
Verify the Sentry SDK behavior — most Sentry SDKs no-op when not 
configured, but some throw. Wrap in try/catch defensively if needed.

- Existing log lines other than auth-fail
- The schema
