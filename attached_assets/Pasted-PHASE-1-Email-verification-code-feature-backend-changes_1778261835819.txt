TASK 3 — New verify endpoint
============================================================

Add new endpoint at the same location as the existing verify-email endpoint in server/replit_integrations/auth/replitAuth.ts:

    app.post("/api/verify-email-code", verifyEmailLimiter, async (req, res) => { ... })

Request body: { email: string, code: string }

Logic (in this order):
1. Validate body shape; 400 if missing/malformed
2. Look up user by email (case-insensitive — existing utility likely exists, find and use it)
3. If user not found → 404 (don't leak whether user exists, but for verify flow it's OK since user just registered)
4. If user.email_verified === true → 200 { already_verified: true } (idempotent — same pattern as existing GET endpoint)
5. If user.verification_code is null → 410 { error: "no_active_code", message: "No active verification code. Request a new one." }
6. If user.verification_code_attempts >= 5 → invalidate code (set verification_code = null), return 429 { error: "too_many_attempts", message: "Too many wrong attempts. Request a new code." }
7. If user.verification_code_expires_at < new Date() → 410 { error: "code_expired", message: "Code expired. Request a new code." }
8. If submitted code !== user.verification_code → increment user.verification_code_attempts, return 401 { error: "invalid_code", remaining_attempts: 5 - new_attempts }
9. On success: set email_verified = true, set verification_code = null, set verification_code_attempts = 0, return 200 { verified: true }

Use the same logging style as the existing verify endpoint (console.warn/info with structured data — see lines 737, 743, 761, 775, 780 for examples).
