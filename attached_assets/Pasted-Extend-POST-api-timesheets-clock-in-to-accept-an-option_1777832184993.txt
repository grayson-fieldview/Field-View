Extend POST /api/timesheets/clock-in to accept an optional source parameter

CONTEXT
Mobile Phase 2 geofencing (Sessions 30-34) is shipping auto clock-in functionality. The mobile client will start sending source: "auto" with clock-in requests when triggered by a geofence enter event. Currently the endpoint's body schema doesn't accept source, which (depending on Zod strictness) will either silently drop it or reject the request with a 400.

Today's existing schema (server/routes.ts around line 1830, established in yesterday's CSRF + projectId coerce session):

const body = z.object({
  projectId: z.coerce.number().int().positive(),
  notes: z.string().max(2000).optional().nullable(),
}).safeParse(req.body ?? {});

CHANGES NEEDED

1. Add source field to the inline Zod body schema:
   source: z.enum(["auto", "manual"]).default("manual")
   
   z.enum() rejects unknown values with a clear error. .default("manual") preserves backward compatibility — existing manual clock-in requests that don't send source still work.

2. Pass body.data.source through to the storage.createTimeEntry call:
   Currently: source: "manual" (hardcoded at L1857-ish)
   New: source: body.data.source

3. Verify time_entries table source column already accepts "auto" and "manual" values. Per the session-22 timesheets spec memory, the column was created as a text/varchar column, not an enum. If it's an enum, may need an ALTER TYPE ADD VALUE migration — flag in diagnosis.
