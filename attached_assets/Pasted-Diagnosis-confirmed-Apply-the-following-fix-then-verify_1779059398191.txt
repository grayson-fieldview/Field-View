ROOT CAUSE
Vercel serverless lambda freezes after res.json() commits. Fire-and-forget Promises in flight at that moment get killed. signed_up and identify on the signup handler reliably lose this race. Other events slip through by happenstance (faster HTTP timing) but it's fragile.

FIX — use @vercel/functions waitUntil to protect all CIO calls

1. Install @vercel/functions:
   pnpm add @vercel/functions

2. Modify server/services/customerio.ts to wrap CIO calls with waitUntil. The key design: do this INSIDE the service module so every existing call site automatically benefits without changes.

Suggested pattern:

import { waitUntil } from "@vercel/functions";

// Safe wrapper — falls back to fire-and-forget in non-Vercel contexts (local dev, scripts)
function deferToVercel(promise: Promise<unknown>): void {
  try {
    waitUntil(promise);
  } catch {
    // Not running in a Vercel function context (local dev) — let it fire-and-forget
    promise.catch(() => {});
  }
}

export async function identifyUser(userId: string, attrs?: Record<string, unknown>): Promise<void> {
  if (!cio) return noopWarn();
  const promise = (async () => {
    try {
      const payload = attrs ?? (await buildAttrs(userId));
      if (!payload) {
        console.warn("[customerio] identify SKIPPED — buildAttrs returned null", { userId });
        return;
      }
      console.log("[customerio] sending attrs:", { userId, payload: JSON.stringify(payload) });
      await cio.identify(userId, payload);
      console.log("[customerio] identify resolved", { userId });
    } catch (err) {
      console.error("[customerio] identify failed:", err);
    }
  })();
  deferToVercel(promise);
}

export async function trackEvent(userId: string, name: CioEventName, data?: Record<string, unknown>): Promise<void> {
  if (!cio) return noopWarn();
  const promise = (async () => {
    try {
      await cio.track(userId, { name, data });
    } catch (err) {
      console.error("[customerio] track failed:", { name, err });
    }
  })();
  deferToVercel(promise);
}

This pattern means:
- In Vercel runtime: waitUntil extends the lambda until the CIO HTTP call completes
- In local dev / scripts / non-Vercel contexts: graceful fallback to plain fire-and-forget
- Existing .catch() patterns at call sites still work but become redundant — they can be cleaned up later
- No changes needed at any call site
