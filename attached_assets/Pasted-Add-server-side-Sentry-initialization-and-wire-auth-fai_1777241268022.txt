## 2. Create server/lib/sentry.ts

Module that initializes Sentry once at server startup and exports the 
Sentry instance.

  import * as Sentry from "@sentry/node";
  
  export function initSentry() {
    const dsn = process.env.SENTRY_DSN;
    if (!dsn) {
      console.warn("[sentry] SENTRY_DSN not set — server-side Sentry disabled");
      return;
    }
    
    Sentry.init({
      dsn,
      environment: process.env.NODE_ENV || "development",
      tracesSampleRate: 1.0,  // capture all transactions for now
      // beforeSend hook can be added later if PII filtering needed
    });
    
    console.log("[sentry] initialized for environment:", process.env.NODE_ENV);
  }
  
  export { Sentry };

If SENTRY_DSN is unset, init() is skipped. Calls to Sentry.captureMessage 
or Sentry.captureException after that will be no-ops (Sentry SDK handles 
this gracefully).

## 4. Wire [auth-fail] to Sentry

In server/replit_integrations/auth/replitAuth.ts, find the logFail 
helper around line 148-156. After the existing console.warn:

  import { Sentry } from "../../lib/sentry";  // adjust path as needed
  
  // ... inside logFail:
  console.warn("[auth-fail]", JSON.stringify({ email, ip, reason, ts }));
  Sentry.captureMessage("Auth failure", {
    level: "warning",
    tags: {
      reason,             // user_not_found / invalid_password / email_not_verified
      auth_provider: "local",
    },
    extra: {
      email,
      ip,
      ts,
    },
  });

Tags are searchable filters in Sentry's UI; extra is metadata visible 
on event detail. DO NOT put email or ip in tags (high cardinality).
