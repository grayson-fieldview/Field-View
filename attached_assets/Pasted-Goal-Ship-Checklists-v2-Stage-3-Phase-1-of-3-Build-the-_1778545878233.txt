    - deleteChecklistTemplateItem(itemId)
    - reorderChecklistTemplateItems(templateId, orderedIds)

C3. Template item options:
    - getChecklistTemplateItemOptions(itemId)
    - createChecklistTemplateItemOption(itemId, label)
    - updateChecklistTemplateItemOption(optionId, patch)
    - deleteChecklistTemplateItemOption(optionId)
    - reorderChecklistTemplateItemOptions(itemId, orderedIds)

C4. Template-level:
    - updateChecklistTemplate(templateId, patch)
      patch: { title?, description? }

C5. INSTANTIATION — the critical one:
    - instantiateChecklistFromTemplate(templateId, projectId, name, 
      createdByUserId, accountId)
    
    Must run inside db.transaction(). Logic:
    1. Look up template; verify template.accountId === accountId, 
       throw if not (account isolation)
    2. Look up template sections, template items, template item options 
       (one query each)
    3. Create the new checklist row in the project (name, projectId, 
       createdById, accountId)
    4. Clone sections in order → build a Map<oldSectionId, newSectionId>
    5. Clone items, preserving label, fieldType, notes, photosRequired, 
       sortOrder. For each item with a non-null template section_id, 
       look up new section id via the Map. (Items with NULL section_id 
       stay NULL.)
    6. For items with fieldType === 'multiple_choice', clone their 
       template item options to the new instance item (preserve label, 
       sortOrder)
    7. Return the new checklist's id
    
    On any failure inside the transaction, everything rolls back — 
    no partial instantiation.
